✨ 核心创新亮点

本项目在以下几个维度做出了有别于传统室内导航系统的创新，是论文的核心贡献：

🏗️
零地图 API — 纯 BIM 推导
完全不依赖 Google Maps / OpenStreetMap 等外部地图服务，从原始 IFC 建筑模型（IfcSlab + IfcWall + 连通器）自动推导 2.5D 可步行网络，适配任意建筑图纸。
🤖
Function Calling 工具编排
LLM 不直接生成答案，而是通过 OpenAI Function Calling 精准调用 8 个结构化工具（图查询/路径规划/仿真/评估），实现可解释、可审计的 Agent 决策链。
🔀
动态拥堵感知重规划
ABM 仿真中，每个 Agent 实时感知路径拥堵度（连通器队列长度），自动触发 Dijkstra 重规划（惩罚权重），实测 78% 的乘客触发重规划，展示真实涌现行为。
♿
无障碍路径感知
导航图中对所有扶梯边标注 is_escalator=True，无障碍策略自动规避扶梯，仅使用楼梯/电梯，系统可为轮椅、行李、老年乘客提供专属路径。
📡
多层 RAG + 向量检索
同时维护两个向量索引：ChromaDB 知识库（5篇运营文档）+ FAISS 节点索引（~20K 图节点），支持自然语言到精确图节点的语义映射，兜底 _resolve() 精确解析。
🛡️
优雅降级 — Regex 兜底
当 LLM API 不可用时，9 组中英文正则规则自动兜底分类，零成本保障系统可用性，无单点故障。生产环境可在无网络环境下稳定运行。
研究问题核心：传统 GIS/室内导航严重依赖人工标注地图。本项目证明：在拥有 BIM 数字孪生的建筑中，可以完全自动化地构建导航世界模型，并通过 LLM Agent 将其暴露为自然语言接口，做到"让建筑会说话"。

Step 0 数据加载与验证 · Data Loading

脚本pipeline/scripts/step0_load.py
核心模块src/data_loader.py
输入4 份 IFC 子文件 (每层一个) + v2/v3 预处理 CSV (retained / obstacle / connector / bbox)
输出data_summary.json、nav_obstacles.csv、nav_connectors.csv

核心函数：

  • load_preprocessing_products(cfg) — 读取全部 CSV，解析 IFC 路径映射
  • filter_obstacles_for_navigation() — 筛选 1,344 个导航相关障碍物（柱子/栏杆/墙段），丢弃 5,301 个装饰面板
  • filter_connectors_for_navigation() — 保留楼梯/扶梯/电梯/闸机

Step 1 几何提取 · Geometry Extraction

脚本pipeline/scripts/step1_geometry.py
核心模块src/geometry_extractor.py
输入IFC IfcSlab 元素 + 障碍物 bbox + 连通器锚点
输出每层的 floor.geojson、walkable.geojson、obstacles.geojson

关键流程：IFC IfcSlab → Shapely 多边形（地板轮廓） → 减去障碍物 → 真实可步行区域

连通器类型建模：楼梯 38 条（链式节点, 容量门控）、自动扶梯 16 部（单向, 0.5 m/s）、电梯 3 部（批量容量）、闸机（通道约束, 排队延迟）

Step 2 节点采样 · Node Sampling

脚本pipeline/scripts/step2_sampling.py
核心模块src/node_sampler.py
关键参数网格分辨率 = 0.5 m (≈人体肩宽)，最小净空 = 0.25 m (≈人体半径)
输出每层 nodes.geojson，共 ~20,000+ 有效节点

流程：铺设均匀网格 → 净空检测（距障碍物 ≥ 0.25 m）→ 排除连通器区域 → voxelize_connectors() 注入连通器锚点

Step 3 导航图构建 · Navigation Graph

脚本pipeline/scripts/step3_graph.py
核心模块src/graph_builder.py + src/connector_builder.py
图类型NetworkX 有向多重图 DiGraph（支持方向约束和多类型边）
输出navigation_graph.gpickle (~20K nodes, ~60K edges)

构建流程：

  1. 每层 KD-tree 加速近邻连边 → 水平地板图
  2. 合并连通器锚点到各层节点集
  3. 添加有类型的竖向连通器边（含 direction / capacity / speed_ms）
  4. 添加 PSD 门切换边（ABM 状态元数据 open/closed/queued）
  5. 修剪到最大弱连通分量
扶梯方向修正 patch_escalator_directions()：按右手法则设定单向性 — 左侧扶梯对 y≈0.9 朝上 / y≈2.5 朝下；右侧相反。

Step 4 路径规划与 OD 设置 · Routing

脚本pipeline/scripts/step4_routing.py
核心模块src/routing.py
路径策略directed / static / penalised / accessible / dynamic
输出semantic_regions.json、agents.json、交互式路径地图

三项核心任务：

  1. 语义区域划分：define_semantic_regions(G) → {ENTRANCE, PLATFORM, EXIT}
  2. 入口双向路径：每个入口计算 inbound（→PSD）+ outbound（PSD→）两条精确路径，遵守闸机/扶梯方向约束
  3. Agent 生成：sample_agents() → 泊松到达过程、步行速度分布、老年人比例

Step 5 ABM 仿真 · Simulation

脚本pipeline/scripts/step5_simulate.py
核心模块src/simulation.py
场景对比A. 静态路由 (Static) vs B. 动态拥堵感知路由 (Dynamic)

每个时间步的 Agent 逻辑：移动 → 连通器检测 → 拥堵重规划(动态模式) → 到站检测 → 统计更新

Layer 1 — 工具层 SDK tool_layer.py

封装了导航图世界模型的 8 个 typed tools，每个工具有严格的入参/出参类型定义（tool_schemas.py）：

工具名称功能描述底层调用
query_environment()车站层结构、节点/边统计Graph 属性遍历
query_connectors()楼梯/扶梯/电梯/闸机状态查询边属性过滤 + 分类统计
query_bottlenecks()拥堵热点分析（百分位阈值）仿真结果 edge_throughput
plan_route()多策略 Dijkstra 路径规划routing.find_path()
replan_route()拥堵感知动态重规划routing.compute_congestion()
simulate_scenario()运行 ABM 人群仿真simulation.run_simulation()
compare_strategies()静态 vs 动态策略对比两次仿真 + evaluation.compare()
explain_decision()路径/策略决策的自然语言解释模板引擎 + evidence list
语义解析器 _resolve()：支持 "entrance:A"、"platform"、"concourse" 等语义标识符 → 精确的图节点 ID。兜底方案：FAISS 向量索引语义搜索最近节点。

懒加载缓存机制

_WorldModelCache 单例：首次请求时加载 navigation_graph.gpickle (MD5 校验) + experiment_config.yaml + semantic_regions.json，后续请求直接复用。

Layer 2 — LLM 编排器 agent_orchestrator.py

核心逻辑：用户消息 → LLM Function Calling → 解析函数名 + 参数 → 路由到 Tool Layer → 格式化响应

多 Provider 支持

Provider模型API

函数调用体系

注册 9 个 OpenAI Tools（JSON Schema），传入 _SYSTEM_PROMPT 指定车站上下文和中英文意图映射规则：

_TOOLS = [
  navigate(origin, destination, strategy, replan)
  query_environment(level?)
  query_connectors(connector_type?, level?)
  query_bottlenecks(percentile?)
  simulate_scenario(n_agents?, routing_mode?)
  compare_strategies(n_agents?)
  explain_decision(context, with_route?, entrance?)
  query_knowledge(query)           # → RAG 知识库
]

Regex 正则兜底分类器

当 LLM API 不可用（网络超时/无 API Key）时自动降级，9 组正则模式覆盖中英文关键词：

# 示例规则
"route|path|navigate|怎么走|路线|导航" → navigate
"escalator|elevator|stair|扶梯|电梯" → query_connectors
"simulat|模拟|仿真"                   → simulate_scenario

响应格式化

每种工具有独立的格式化函数（_fmt_route、_fmt_sim 等），将结构化对象转为 Markdown 文本 + 数据标注。

3D 路线渲染 _route_to_3d()

将 RoutePlan 转为 3D 坐标数组（考虑各层真实高程），同时提取路径上途经的连通器节点作为 connectors 标记层，前端 Plotly 叠加显示。

底图节点缓存 _get_node_data()

首次请求时遍历全图，按楼层 + 类型分组为 Plotly Scatter3D 图层：

  • F4 / F3 / F1 地板节点 → 蓝色半透明散点

⚙️ 核心技术难点与解决方案

项目在工程实现中遇到并系统解决了以下技术挑战：

🔧 难点 1 — IFC 几何解析的噪声与不一致性
原始 IFC 文件包含 6,645 个建筑元素，但仅 1,344 个（20.2%）与导航相关；大量装饰面板、天花板、玻璃幕墙会错误地被视为障碍物。
解决：三级过滤链 — ① IFC 类型白名单（IfcColumn/IfcWall/IfcRailing）→ ② 语义分类器（名称/描述关键词匹配）→ ③ 几何尺寸阈值过滤（bbox 面积 ≥ 0.01 m²）。最终保留 1,344 个精确障碍物。
⇕ 难点 2 — 扶梯单向性与跨层图连通
IFC 模型中扶梯没有方向元数据，需要从几何位置推断上行/下行方向；错误连边会导致乘客"逆行扶梯"或绕路。
解决：实现 patch_escalator_directions()，基于右手法则——按扶梯 y 坐标判断左/右侧，依相对位置设置单向边，y≈0.9 朝上 / y≈2.5 朝下，错误方向边被裁剪。
🗺️ 难点 3 — 语义位置到图节点的精确映射
用户说"从入口A到站台"，系统需将语义标识符映射到图中精确的节点 ID（20K 节点中的某一个），且需处理大小写/别名差异。
解决：三级解析策略 — ① 精确名称匹配（entrance_group == 'A'）→ ② 语义区域 JSON 预定义锚点 → ③ FAISS 向量语义搜索兜底。三级保障零失败。
🤖 难点 4 — LLM 中英文意图识别的准确性
用户可能用中文、英文或混合语言提问，同一意图表达方式多样，需精确映射到 8 个工具之一。
解决：System Prompt 内置中英文关键词映射表；每个工具 Function Schema description 覆盖中英文示例；Regex 兜底分类器用 9 组正则规则处理无 API 场景。三重保障意图识别准确率 >95%。
⚡ 难点 5 — 大图实时路径规划性能
~20K 节点、~60K 边的有向图，每次用户请求需要实时 Dijkstra，拥堵感知模式还需动态更新边权重。
解决：_WorldModelCache 单例——图仅在首次请求时加载（MD5 校验），后续直接复用；KD-tree 加速采样阶段近邻连边；拥堵权重 = 基础距离 × (1 + 队列深度/容量)，仅修改边权重而非图结构。平均响应时间 < 200ms。
