02 Tool Layer SDK

Seven typed tools are registered as JSON schemas for DeepSeek function calling.

query_environment(floor?, region?)
Returns current agent counts, congestion levels, and region status for a given floor or semantic region. Used for "how busy is …" queries.
floor: F1|F2|F3|F4 region: str
plan_route(origin, destination, mode?)
Runs Dijkstra from the nearest named node to destination; returns path node list, total time, and floor transitions. Mode "accessible" restricts to lifts only.
origin: str destination: str mode: "normal"|"accessible"|"fast"
simulate_scenario(scenario)
Triggers or retrieves results of a named simulation scenario (A/B/C). Returns arrival rate, avg travel time, and peak congestion.
scenario: "static"|"dynamic"|"wheelchair"
get_congestion(area?, threshold?)
Returns edges/nodes exceeding a congestion threshold. Optionally filtered to a floor or named area. Used for "which route is least congested" queries.
area: str threshold: 0.0–1.0
find_connector(type?, floor?, accessible?)
Locates the nearest connector of the given type from a reference location. Supports filtering by accessible (lifts only for wheelchair users).
type: "stair"|"escalator"|"lift" floor: str accessible: bool
get_accessibility_path(origin, destination)
Specialised version of plan_route with lift-only constraint and wheelchair speed coefficients. Returns accessible route with wait-time estimates.
origin: str destination: str
list_semantic_regions(floor?)
Returns all named semantic regions on a given floor (or all floors). Enables the agent to ground vague user terms like "exit" or "platform" to specific nodes.
floor: str (optional)
_WorldModelCache — Lazy Loading The Tool Layer holds a singleton _WorldModelCache that lazily loads the NavigationGraph, experiment config, and semantic regions on first call. Subsequent tool calls reuse the cached objects — no re-serialisation overhead.

03 Dual-Engine Intent Recognition

Primary: DeepSeek Function Calling. Fallback: 9 compiled Regex rules for zero-loss offline operation.

🧠 DeepSeek Function Calling

  • Full JSON schema registered for all 7 tools
  • Structured parameter extraction from free text
  • Multi-turn conversation context (last 4 turns)
  • Handles ambiguous phrasings, mixed EN/CN input
  • Returns structured tool call + human response

🔤 Regex Fallback Classifier

  • 9 compiled regex rules (bilingual EN + CN)
  • Covers: route, congestion, accessibility, environment, scenario, connector
  • Zero additional latency — pure in-process matching
  • Graceful degradation: no API key required
  • Identical tool dispatch interface as primary engine

05 FAISS Semantic Node Index

A FAISS flat index over node semantic labels allows fuzzy spatial grounding — "ticket gate" → nearest node ID.

How It Works

1Node label embedding

Each node is assigned a human-readable semantic label (e.g., "platform edge escalator top", "main exit ticket gate"). Labels are embedded to 384-dim vectors.

2FAISS flat L2 index

All 18,648 node vectors are added to a FAISS IndexFlatL2. Stored as faiss_node_index.bin + node_label_map.json.

3Query-time grounding

When the agent receives an entity mention (e.g., "nearest wheelchair lift"), the FAISS index returns the top-5 matching node IDs, which are passed to plan_route as candidate origins/destinations.

Dual-Index Architecture ChromaDB (document chunks) + FAISS (graph nodes) operate in parallel: ChromaDB answers "what is X"-style questions; FAISS resolves "where is X"-style spatial grounding. Both indices are queried in every non-trivial LLM turn and their results merged into the system prompt context.

07 Q&A Examples

Sample natural-language queries and the tools they invoke.

How do I get from the platform to Exit B using a wheelchair?
The accessible route from platform (F1) to Exit B (F3) uses Lift 02 near the centre of the platform. Estimated time: 4 min 12 sec. The lift is currently unoccupied (0 of 8 capacity).
get_accessibility_path(origin="platform_centre", destination="exit_B") → find_connector(type="lift")
Which exit is least congested right now?
Exit A (F4, north) has the lowest congestion factor at 0.12. Exit C (0.41) and Exit B (0.67) are busier. I recommend Exit A via the east escalator bank.
get_congestion(area="exits") → query_environment(floor="F4")
现在站台有多少人？
目前站台层（F1）有约 47 名乘客，集中在中央候车区。拥挤系数为 0.23，属于正常水平。
query_environment(floor="F1", region="platform")
Show me the dynamic simulation results after the train arrives.
Scenario B (Dynamic): after train arrival at tick 120, 312 re-plan events were triggered. Average travel time increased to 178.4 s (+11% vs. baseline). Peak congestion was 0.78 near the main staircase.
simulate_scenario(scenario="dynamic")
Plan the fastest route from the ticket gate to the eastbound platform.
Fastest route: Ticket Gate → Main Concourse → Escalator E2 (down) → Platform F1 East. Estimated time: 2 min 8 sec (128 s). 3 floor transitions via 1 escalator.
plan_route(origin="ticket_gate", destination="platform_east", mode="fast")
